1 The 9-Layer Architecture

Every layer is designed so future phases ADD to it — nothing needs to be rewritten. Layers 1–6 and 9 exist today. Layers 7–8 and the parallel workflow layer are designed and sequenced but not yet built.

2 Product Vision — The Agent Marketplace

The end-state product is a multi-tenant SaaS platform where any organisation subscribes to a sidebar of department agents. Each agent operates in complete data isolation from every other tenant. HR is live now; Finance, Legal, and Admin follow in Phase 2.

3 Security Model — Defence in Depth

Four independent enforcement layers surround every piece of data. A failure in any single layer does not compromise the system — the other three still hold. Three more layers are added in Phase 2 and 3.

flowchart TD REQ(["Request arrives"]) --> L1 subgraph L1["LAYER 1 — Policy Before Prompt (built)"] T1["get_specs_for_role\nLLM only receives tool specs for its role\nIt cannot call — or even attempt — what it cannot see"] end L1 --> L2 subgraph L2["LAYER 2 — Execute Authorization (built)"] T2["registry.execute — second role check\nIndependent of the LLM call\nBlocks disallowed tools before any code runs"] end L2 --> L3 subgraph L3["LAYER 3 — Tenant Row Isolation (built)"] T3["WHERE tenant_id = x on every SQL query\nOther tenants data unreachable at the DB level\nNo application code path can skip this"] end L3 --> L4 subgraph L4["LAYER 4 — Immutable Audit Trail (built)"] T4["INSERT audit_log on every execution\nActor, role, tool, input, outcome, latency — always\nHash-chain + WORM mirror added Phase 3"] end L4 --> DATA[("Protected Data")] subgraph FUTURE["Phase 2-3 additions — defence in depth continues"] F1["RLS FORCE\nDatabase enforces tenant isolation\neven if app code has a bug (Phase 1.5)"] F2["Field encryption\nSalary, national_id encrypted at rest\nBlind index pattern (Phase 3)"] F3["Hash-chained audit\nTamper-evident row chain\nWORM mirror for compliance (Phase 3)"] end style L1 fill:#1e2a5e,stroke:#1e40af style L2 fill:#3b1e5e,stroke:#6d28d9 style L3 fill:#1e3a2f,stroke:#166534 style L4 fill:#5e3d1e,stroke:#92400e style FUTURE fill:#111420,stroke:#252b42

4 Phased Roadmap

Each phase is a prerequisite for the next. Auth before writes. RLS before RAG. The architecture is already designed for all phases — they add to it, they don't change it.

  • LLMProvider abstraction
  • ToolRegistry gateway
  • ToolContext
  • audit_log table

  • Session history

5 From Request to Compliance

Every action — even a simple "generate salary cert" — produces four traceable outputs: the document, two audit log entries, a session history update, and a document record. Nothing happens that isn't logged.
