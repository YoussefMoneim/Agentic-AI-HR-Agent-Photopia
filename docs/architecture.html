1 System Architecture Overview

Standard three-tier web application. The browser talks to a Python API; the API calls an LLM to interpret requests and decide what tools to run; all persistent data lives in PostgreSQL. The LLM provider is swappable — Claude (Anthropic) or Grok (xAI) — configured by a single environment variable.

2 End-to-End Request Flow

The exact path of every chat message. The orchestrator may loop through the LLM and tools multiple times — the LLM calls a tool, sees the result, calls another, and so on — until it has enough information to produce a final text answer.

3 ReAct Agent Loop

ReAct = Reason + Act. The LLM is stateless — it has no memory and cannot run tools itself. The orchestrator gives it memory (by managing the messages list) and agency (by executing the tools it requests). The hard cap of 10 iterations prevents runaway API costs.

4 Tool Catalog & Role Access

The agent has 10 tools across two categories. Role-based access operates at two independent layers: the LLM only receives specs for tools its role permits (it cannot even attempt others), and the registry performs a second check before any tool actually runs.

5 Database Schema

PostgreSQL with four tables. Every row in users, employees, and audit_log is scoped to a tenant_id UUID — this is the architectural foundation of multi-tenant isolation. No query can reach another company's data without knowing their UUID.

6 Multi-Tenant Data Isolation

One database serves multiple companies simultaneously. A tenant_id UUID is a foreign key on every data table. Every SQL query in the codebase includes WHERE tenant_id = %s — it is architecturally impossible for Fotopia to read Acme Corp's data through the same API.

7 Security Architecture

Defence in depth — four independent layers between a request and the data. A failure in any single layer does not compromise the system because the remaining three still enforce access control independently.

8 Frontend Component Tree

React SPA built with Vite. All chat state lives in ChatInterface. MessageBubble routes agent responses through ReactMarkdown so tables, bold text, lists, and code blocks all render correctly — user messages stay as plain text.

Phase 1 — Demo (Current)

  • Hardcoded user: Nourhan Hosny (hr_manager)
  • Tenant resolved from TENANT_SLUG env var
  • Conversation history in-memory (session dict)
  • LLM swappable: Claude or Grok via env var
  • 3 demo employees seeded in database

Phase 2 — Production

  • JWT authentication per request
  • Tenant + role resolved from token claims
  • Redis for session storage (survives restarts)
  • Managed cloud DB: AWS RDS or Supabase
  • Finance, Legal, and Admin agents live
  • Employee self-service portal
