What It Does

Plain-language overview — what the agent handles, why it matters, and what makes it enterprise-ready.

HR managers spend 40–60% of their time on requests employees could resolve themselves in seconds.

The Fotopia HR Agent handles those requests — via chat or plain email — with the same policy precision as a senior HR officer. Every action is logged. Humans approve anything consequential. The AI never makes final decisions.

What's Live Today

  • 📅
    Leave Management — full lifecycle Submit, check balance, approve, reject, cancel, track status. Enforces WIN Holding HR/BTE 001/7-2025 policy automatically — notice periods, balance checks, team availability, probation restrictions.
  • 💋
    Bidirectional Email Employee emails hr.agent.fotopia@gmail.com in plain language. Claude Haiku reads it, extracts intent and dates, submits the request, notifies the manager, and confirms back to the employee's inbox — no UI required.
  • 📄
    Official Document Generation Salary certificates, experience certificates, To Whom It May Concern letters — generated in seconds. Bilingual (Arabic + English). DB-verified data. Every document logged with a reference code (SC-/TW-/EC-).
  • 📈
    Odoo ERP Sync Every approved leave is automatically created in Odoo via XML-RPC. No double entry. Managers see approvals reflected in the ERP in real time.
  • 📜
    Policy Q&A Employees ask "can I take leave during probation?" or "how many days for hajj?" — agent searches the actual policy document and cites the exact section. No hallucinated answers.
  • 🧮
    End-of-Service Calculator Deterministic Egyptian Labor Law gratuity calculation — broken down by period, cap applied. LLM never touches the math; all Python.

Why It's Enterprise-Ready

  • 📋
    Every action audited The audit log records every tool call — success, denial, error — with actor, role, latency, and result. No tool can bypass this. Phase 3 adds hash-chain tamper evidence for regulated environments.
  • 👤
    Human always approves consequential actions Leave approvals need a manager. Employee record creation and social insurance enrollment are permanently HITL-gated — the AI can never execute these unilaterally.
  • 🏢
    Multi-tenant from day 1 Every table carries tenant_id. One deployment serves multiple clients. Data is physically isolated at the database row level. Phase 2 adds Row-Level Security FORCE at the Postgres engine level.
  • 🔧
    LLM is swappable Claude is the default. Grok is a drop-in via one env var. Self-hosted Llama/Qwen works for clients who can't send data to a cloud API (Egyptian PDPL concern). Zero code changes required.
  • 🔒
    Access control lives in code, not prompts Role checks happen in the ToolRegistry (Python) and PostgreSQL (RLS) — not in the system prompt. A jailbreak cannot override a database permission check.
  • 🇬🇧
    Egyptian legal compliance built in WIN Holding HR/BTE 001/7-2025 policy enforced at every leave submission. End-of-service follows Egyptian Labor Law Articles 117–119. Egypt PDPL cross-border transfer is flagged as a go-live prerequisite — demo runs on synthetic data.

Key Design Principles

🔒

Policy Before Prompt

ToolRegistry filters available tools by role before Claude sees them. An employee cannot invoke payroll tools because they are never offered to the LLM — not because of a prompt instruction.

📋

Immutable Audit Trail

Every tool execution — success, failure, or denial — writes to audit_log. No tool can bypass this; the ToolRegistry is the only allowed call path.

🧮

LLM Never Does Math

Leave duration, working day count, EOS gratuity — all deterministic Python. The LLM extracts intent; business logic is code. A wrong number on a legal document is a lawsuit.

🔄

Three Approval Channels — One Handler

Manager approves via email reply, link click, or UI inbox. All three paths converge to the same /api/leave/resolve handler with identical authorization logic.

Security Architecture

Defense-in-depth across three dimensions: data in transit, data at rest, and role-based access control.

Data in Transit

🔒

TLS 1.2+ on all external calls

Anthropic API (HTTPS 443), Odoo XML-RPC (HTTPS 443), Azure Communication Services (HTTPS 443). No plaintext external channels.

📧

SMTP STARTTLS — Office 365

Outbound email uses STARTTLS upgrade on port 587. TLS is negotiated before credentials are transmitted. Connection is IPv4-forced to prevent Docker IPv6 routing failures.

📬

IMAP over SSL — Gmail

Inbound approval listener connects via SSL on port 993. An App Password (not the account password) is used — limits credential scope to IMAP access only.

🐳

Docker network isolation

Backend and database communicate over an internal Docker bridge network. The database port (5432) is not exposed to the host in production — only the backend reaches it.

🎫

Signed one-time approval tokens

Each approval link contains a token stored in pending_actions. Tokens are consumed on first use — prevents replay attacks on the leave approval endpoint.

Data at Rest

📐

PostgreSQL Row-Level Security

tenant_id on every table. Phase 2 enables ENABLE ROW LEVEL SECURITY FORCE — queries from any role are automatically scoped to their tenant. Cross-tenant data leaks become impossible at the database engine level.

RLS FORCE targeted for Phase 2

🔑

Secrets via environment variables only

All credentials (API keys, DB passwords, SMTP) loaded from .env via config.py. No secret appears in source code or version control. .env is gitignored.

🔐

Field-level encryption — Phase 3

national_id and salary columns encrypted at rest with a blind index pattern — encrypted storage, searchable via deterministic hash, plaintext never written to disk.

📜

Immutable audit log

Every tool call logged by the ToolRegistry — cannot be bypassed. Phase 3 adds a hash-chain (each row hashes the previous row's digest) and WORM-mirrored storage for tamper evidence in regulated environments.

Role-Based Access Control

Four roles. Access enforced at two independent layers: ToolRegistry (application) and PostgreSQL RLS (database). Both must pass.

Role Read own record Read all employees Submit leave Approve leave Generate documents EOS calculation Admin tools
employee ✅❌✅❌❌❌❌
hr_staff ✅✅✅❌✅✅❌
hr_manager ✅✅✅✅✅✅❌
admin ✅✅✅✅✅✅✅

Row-level: HR roles see all employees in the tenant. An employee sees only their own record, matched via employee_code in ToolContext — enforced in every tool via _can_access_employee().

Hard Security Rules — Never Violated

⛔

No SDK import outside llm/claude.py

Anthropic SDK imported in exactly one file. Prevents prompt injection via rogue tool or middleware.

⛔

No direct DB queries in tools

All data access via the DataSource abstraction. Row-Level Security cannot be bypassed by a raw connection from a tool.

⛔

No access control in the system prompt

RBAC lives in tools + RLS. A jailbreak cannot override a database permission check — it is not a text instruction.

⛔

LLM never does math

Leave durations, EOS gratuity — all deterministic Python. A wrong number on a legal document is a lawsuit.

⛔

Uploaded documents are data, not instructions

Resumes and certificates are OCR-extracted to structured fields. Raw uploaded content never enters a context that can trigger tool calls.

⛔

No live self-modifying prompts in production

Offline prompt optimization (DSPy) produces frozen, version-controlled, human-reviewed artifacts only.

Email Architecture Summary

📤

Outbound — Office 365 SMTP

Sent FROM fotoagent@fotopiatech.com. The Reply-To header is set to the Gmail IMAP address so manager replies land in the listener inbox, not the Office 365 inbox.

📥

Inbound — Gmail IMAP listener

Polls hr.agent.fotopia@gmail.com every 3 seconds. Handles two message types: (1) employee leave requests by email — classified by Claude Haiku; (2) manager APPROVE/REJECT replies.

👤

Delivery target (notification_email)

Each employee has notification_email decoupled from employees.email. Confirmations go to the personal inbox; the work email is the Odoo/identity lookup key.
