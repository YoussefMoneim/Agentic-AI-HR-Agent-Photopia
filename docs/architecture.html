Key Design Principles

🔒

Policy Before Prompt

ToolRegistry filters available tools by role before Claude sees them. An employee cannot invoke payroll tools because they are never offered to the LLM.

📋

Immutable Audit Trail

Every tool execution — success, failure, or denial — writes to audit_log. No tool can bypass this; the ToolRegistry is the only allowed call path.

🧮

LLM Never Does Math

Leave duration is calculated in deterministic Python (weekday counter). The LLM extracts intent; business logic is code, not inference.

🔄

Triple Approval Channel

Manager approves via email reply, link click, or UI inbox. All three paths converge to the same /api/leave/resolve handler with identical authorization logic.

Security Architecture

Defense-in-depth across three dimensions: data in transit, data at rest, and role-based access control.

Data in Transit

🔒

TLS 1.2+ on all external calls

Anthropic API (HTTPS 443), Odoo XML-RPC (HTTPS 443), Azure Communication Services (HTTPS 443). No plaintext external channels.

📧

SMTP STARTTLS — Office 365

Outbound email uses STARTTLS upgrade on port 587. TLS is negotiated before any credentials are transmitted. Connection is IPv4-forced to prevent Docker IPv6 routing failures.

📬

IMAP over SSL — Gmail

Inbound approval listener connects via SSL on port 993. An App Password (not the account password) is used — limits scope of credential to IMAP access only.

🐳

Docker network isolation

Backend and database communicate over an internal Docker bridge network. The database port (5432) is not exposed to the host in production — only the backend reaches it.

🎫

Signed one-time approval tokens

Each approval link contains a token stored in pending_actions. Tokens are consumed on first use. Prevents replay attacks on the leave approval endpoint.

Data at Rest

📐

PostgreSQL Row-Level Security

tenant_id on every table. Phase 2 enables ALTER TABLE ... ENABLE ROW LEVEL SECURITY ... FORCE. Queries from any role are automatically scoped to their tenant — cross-tenant data leaks become impossible at the database engine level.

RLS FORCE targeted for Phase 2

🔑

Secrets via environment variables only

All credentials (API keys, DB passwords, SMTP) are loaded from .env via config.py. No secret appears in source code or version control. .env is gitignored.

🔐

Field-level encryption — Phase 3

national_id and salary columns encrypted at rest with a blind index pattern — encrypted storage, searchable via deterministic hash, plaintext never written to disk.

📜

Immutable audit log

Every tool call logged by the ToolRegistry — cannot be bypassed. Phase 3 adds a hash-chain (each row hashes the previous row's digest) and WORM-mirrored storage for tamper evidence in regulated environments.

Role-Based Access Control

Four roles. Access is enforced at two independent layers: ToolRegistry (application) and PostgreSQL RLS (database). Both must pass.

Role Read own record Read all employees Submit leave Approve leave Generate documents EOS calculation Admin tools
employee ✅❌✅❌❌❌❌
hr_staff ✅✅✅❌✅✅❌
hr_manager ✅✅✅✅✅✅❌

Row-level: HR roles see all employees in the tenant. An employee can only access their own record, matched via employee_code in ToolContext — enforced in every tool via _can_access_employee().

Hard Security Rules

⛔

No SDK import outside llm/claude.py

Anthropic SDK imported in exactly one file. Prevents prompt injection via rogue tool or middleware files.

⛔

No direct DB queries in tools

All data access via the DataSource abstraction. Row-Level Security cannot be bypassed by a raw connection from a tool.

⛔

No access control in the system prompt

RBAC lives in tools + RLS. A jailbreak cannot override database-enforced permissions — the permission check is not a text instruction.

⛔

LLM never does math

Leave durations, EOS gratuity calculations — all deterministic Python. A wrong number on a legal document is a lawsuit.

⛔

Uploaded documents are data, not instructions

Resumes and certificates are OCR-extracted to structured fields. Raw uploaded content never enters a context that can trigger tool calls.

⛔

No live self-modifying prompts in production

Offline prompt optimization (DSPy) produces frozen, version-controlled, human-reviewed artifacts. Live optimizer access to RBAC prompts is prohibited.

Email Architecture Summary

📤

Outbound — Office 365 SMTP

Sent FROM fotoagent@fotopiatech.com. The Reply-To header is set to the Gmail IMAP address so manager replies land in the listener inbox, not the Office 365 inbox.

📥

Inbound — Gmail IMAP listener

Polls hr.agent.fotopia@gmail.com every 3 seconds. Parses APPROVE/REJECT replies from managers and routes to /api/leave/resolve.

👤

Delivery target (notification_email)

Each employee has a notification_email field decoupled from employees.email. The former is the delivery address; the latter is the Odoo/identity lookup key.
